1. Contexte de la mission

Dans le cadre du cours de sécurité réseau animé par Faroyan Malchaz, cette mission d'audit avait pour objectif d'analyser un serveur cible (192.168.56.101) sur un segment réseau isolé, d'identifier ses vulnérabilités, et d'y apporter une remédiation concrète. La mission couvrait quatre phases évaluées : reconnaissance, audit profond, analyse documentée et remédiation.

PHASE 01

Reconnaissance

Analyse locale des ports avec ss -tunlp et identification de l'adresse IP cible.

4 points
PHASE 02

Audit Profond

Démasquage des services réels via scan de version Nmap (-sV) et confirmation Netcat.

6 points
PHASE 03

Analyse & Veille

Justification du danger des services détectés, appuyée sur ANSSI, OWASP et CVE MITRE.

4 points
PHASE 04

Remédiation

Filtrage réseau via règle Windows Defender Firewall, validé par re-scan Nmap.

6 points

2. Reconnaissance - Cartographie des ports

3. Audit profond - Démasquage des services

La sécurité par l'obscurité consiste à masquer des services sur des ports non standard. L'outil Nmap avec le flag -sV permet de démasquer la véritable identité de chaque service, indépendamment du port utilisé. Deux anomalies majeures ont été identifiées.

attaquant@kali:~$ 
nmap -sV -T4 192.168.56.101

PORT    STATE SERVICE VERSION
21/tcp  open  ftp     vsftpd 3.0.5
22/tcp  open  ssh     OpenSSH 10.0p2
30/tcp  open  http    Apache httpd 2.4.66   ← Devrait être sur 80/443
80/tcp  open  telnet  Linux telnetd         ← Port HTTP utilisé par Telnet !
110/tcp open  pop3    Dovecot pop3d
139/tcp open  netbios Samba smbd 4
143/tcp open  imap    Dovecot imapd
445/tcp open  smb     Samba smbd 4

nc -v 192.168.56.101 30
✓ Apache2 Debian Default Page - It works!

curl http://192.168.56.101:30
✓ Réponse HTTP 200 confirmée - Apache 2.4.66 sur port 30

4. Analyse des vulnérabilités

Le service le plus critique est Telnet tournant sur le port 80. Cette configuration cumule trois risques majeurs : transmission de credentials en clair (sniffable), contournement des pare-feux filtrant le port 23, et discrétion pour un attaquant souhaitant maintenir un accès persistant sans déclencher les IDS. Apache sur le port 30 présente également un risque d'invisibilité aux scans basiques.

ANSSI

Guide Hygiène Informatique - Règle 15 : Chiffrer les communications sensibles

OWASP

A02:2021 - Cryptographic Failures : utilisation de protocoles non chiffrés

CVE MITRE

CWE-312 : Cleartext Storage / CWE-319 : Cleartext Transmission

RFC 854

Telnet Protocol Specification - protocole conçu sans aucune sécurité (1983)

5. Remédiation - Blocage pare-feu

Sans droits root sur la VM cible, la remédiation a été appliquée au niveau du pare-feu Windows de la machine hôte via Windows Defender Firewall avec fonctions avancées. Une règle de trafic sortant bloque les connexions vers le port 80 de la cible.

  • Ouverture de Windows Defender Firewall → Paramètres avancés
  • Création d'une règle de trafic sortant : Type Port → TCP → Port 80
  • Action : Bloquer la connexion - Profils : Domaine, Privé, Public
  • Nom de la règle : Blocage Telnet port 80
attaquant@kali:~$ - Validation post-remédiation 
nmap -sV -T4 192.168.56.101

PORT    STATE  SERVICE VERSION
21/tcp  open   ftp     vsftpd 3.0.5
22/tcp  open   ssh     OpenSSH 10.0p2
30/tcp  open   http    Apache httpd 2.4.66
80/tcp  filtered -   ← Port 80 filtré, Telnet inaccessible ✓
110/tcp open   pop3    Dovecot pop3d

Le port 80 n'apparaît plus comme open dans les résultats Nmap - il apparaît désormais filtered, confirmant que la règle pare-feu est active. La connexion Telnet sur ce port est bloquée avec succès.

  • Recommandation 1 : Remplacer Telnet par SSH (port 22) - protocole chiffré
  • Recommandation 2 : Reconfigurer Apache sur le port 443 avec certificat SSL/TLS
  • Recommandation 3 : Mettre en place une règle firewall permanente sur la VM elle-même
  • Recommandation 4 : Réaliser des audits réguliers avec nmap -sV pour détecter toute anomalie
